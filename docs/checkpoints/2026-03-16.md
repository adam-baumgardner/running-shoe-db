# Checkpoint: March 16, 2026

## Purpose

This checkpoint captures the state of `codex/rebuild-foundation` after the first major end-to-end product build. It is intended to support stakeholder review and a merge decision on whether this branch should become the new default baseline.

## What is working on the branch

- Next.js + TypeScript rebuild has fully replaced the Django prototype.
- Canonical shoe modeling now separates:
  - brand
  - shoe family
  - release/version
  - release-specific specs
- Public product flows exist for:
  - catalog browsing and filtering
  - parent shoe-family pages
  - release-specific detail pages
  - release-aware comparison
  - release history and version navigation
  - release-to-release "What Changed" summaries
- Review intelligence exists for:
  - source-backed review cards
  - weighted release-level consensus
  - reconciled cross-source themes
  - AI-generated release summaries
  - editorial override of takeaways and AI summary
- Internal/editorial tooling exists for:
  - manual brand, shoe, release, and source entry
  - review moderation
  - duplicate marking
  - override history
  - crawl triggering
  - crawl cadence controls
- Automated ingestion is implemented for:
  - Believe in the Run
  - Reddit `r/RunningShoeGeeks`
  - RunRepeat
  - Doctors of Running

## What is not ready yet

- Production deployment reliability is not stable enough.
- `/internal` still needs a dedicated stabilization pass in deployed environments.
- Internal auth is still shared HTTP Basic Auth, not proper user-level authentication.
- Visual polish is incomplete.
- Automated spec extraction is still conservative and should improve over time.
- Test coverage is still light relative to product complexity.

## Risks to resolve before merging to `main`

- Vercel/Supabase runtime behavior needs a focused operational debugging pass.
- The branch now contains a large amount of product scope, so merge risk is mainly operational rather than architectural.
- Documentation and deploy runbooks should stay current as the system stabilizes.

## Recommendation

This is a good checkpoint for stakeholder review.

Recommended sequence:

1. Review the branch locally as the new baseline product.
2. Decide whether `codex/rebuild-foundation` should replace the old `main`.
3. If yes, do one short hardening pass before merge:
   - deployment stabilization
   - minimal UI cleanup
   - explicit merge note / migration note
4. Merge to `main` only after the deployed baseline is predictable again.

## Suggested next workstreams

### Track 1: baseline hardening

- fix production/internal deployment reliability
- add a minimal smoke-test path for key pages
- replace shared internal auth later with proper auth

### Track 2: product quality

- polish parent/release page layout
- deepen spec extraction quality
- improve review/source freshness signaling

### Track 3: content scale

- expand release coverage
- improve crawl scheduling heuristics
- add more source-specific extraction depth
